# M5 分支后续功能规划

本文档统一记录两类后续工作：

- 用户希望在 M5 分支新增的多地点连接配置能力；
- 原作者上游项目更新后，适合本 M5 分支选择性跟进的功能。

原则：优先保持 M5 Core / M5 Fire 已验证可用的显示、三按键、音频、LED 提示逻辑稳定；上游代码只做选择性移植，不直接整体合并。

## 一、计划新增功能：多地点连接配置

### 目标

让同一台 M5 可以带到不同位置使用，每个位置保存一组 WiFi 与 FMO 地址。用户只需切换当前配置，就可以连接对应地点的网络和 FMO 主机。

推荐把它做成“连接配置槽”：

| 配置项 | 内容 |
| --- | --- |
| 配置名称 | 例如“家里”“车上”“基地台”“临时” |
| WiFi SSID | 当前地点的 WiFi 名称 |
| WiFi 密码 | 当前地点的 WiFi 密码 |
| FMO Host | IP、DDNS 域名，或域名加端口 |

建议第一版提供 5 个配置槽。配置 1 继承当前默认值和旧设置，其余配置为空。

### FMO 地址格式

当前代码的 `fmo_host` 字段已经按 Host 生成 WebSocket URL，理论上支持 `example.com:8080` 这种形式。后续需要把界面和校验补齐。

建议支持：

| 输入 | 说明 |
| --- | --- |
| `192.168.31.146` | 局域网 IP，默认端口 |
| `192.168.31.146:8080` | 局域网 IP + 端口 |
| `fmo.example.com` | DDNS / 域名，默认端口 |
| `fmo.example.com:8080` | DDNS / 域名 + 端口 |

建议不允许用户输入完整 URL，例如 `http://...`、`ws://...` 或带路径的 `/audio`。设置页只填写 `Host[:Port]`，程序继续自动生成：

```text
ws://<host>/audio
ws://<host>/events
ws://<host>/ws
```

### 数据结构建议

在 `app_settings_t` 中新增连接配置数组和当前激活序号：

```c
#define APP_CONNECTION_PROFILE_MAX 5
#define APP_PROFILE_NAME_MAX       16
#define APP_FMO_HOST_MAX           96

typedef struct {
    char name[APP_PROFILE_NAME_MAX];
    char wifi_ssid[33];
    char wifi_password[64];
    char fmo_host[APP_FMO_HOST_MAX];
} app_connection_profile_t;
```

运行时继续保留 `ws_audio_url`、`ws_event_url`、`ws_station_url` 这三个派生 URL，但它们应由当前激活配置的 `fmo_host` 生成。

需要注意：

- 提升 `APP_SETTINGS_VERSION`；
- 尽量迁移旧 NVS 中的单 WiFi / 单 FMO 设置到配置 1；
- 旧字段如果继续保留，应明确它们只是当前激活配置的兼容镜像；
- 如果直接移除旧字段，需要同步修改 `wifi_manager.c`、`audio_ws.c`、`app_ui.c` 等所有读取点。

### 设置界面建议

设置首页新增一行：

```text
连接配置    1/5 家里
```

按键建议：

| 操作 | 行为 |
| --- | --- |
| A / C | 在设置菜单中移动焦点 |
| B | 进入连接配置页 |
| 连接配置页 A / C | 切换配置槽 |
| 连接配置页 B | 选中当前配置并返回，必要时重连 |

保留现有的“FMO 地址”和“WiFi 设置”页面，但它们只编辑当前激活配置。页面标题建议显示当前配置名，例如：

```text
FMO 地址  配置 1
WiFi 设置  配置 1
```

第一版可以暂时不做复杂的配置重命名。如果要支持名称编辑，可在第二阶段增加“配置名称”输入页。

### 连接切换行为

切换当前配置后建议立即执行：

1. 保存当前配置序号；
2. 停止 WebSocket 音频 / 事件连接；
3. 停止或重启 WiFi；
4. 使用新配置的 SSID / 密码重新连接；
5. WiFi 获取 IP 后按现有流程连接 FMO。

如果用户只保存 FMO 地址但 WiFi 不变，可以只重启 WebSocket；如果保存 WiFi，则需要重启 WiFi。第一版为了可靠，可以统一提示“保存重启”或统一执行 WiFi 重连。

### 验证重点

- 旧设备升级后，原 WiFi 和 FMO 地址不能丢失；
- 配置 1 默认 FMO 地址仍为 `192.168.31.146`；
- DDNS + 端口生成的三个 WebSocket URL 正确；
- 空配置槽不能误触发连接；
- 切换配置后 UI 显示、WiFi 状态、FMO 状态一致；
- 节能模式恢复后仍使用当前激活配置；
- LED 通联提示逻辑不受连接配置改动影响。

## 二、上游更新可跟进功能

上游当前新增内容主要包括蜂鸣器、QTH 地址显示、电池显示方式选择。M5 分支建议选择性跟进。

### 1. 电池显示方式选择

建议跟进，优先级高。

上游新增了电池显示模式，可在图标、百分比、电压之间切换。这个功能适合 M5 分支，因为 M5 屏幕右上角本来就有电池状态，用户也可能需要更直观地看电压。

M5 分支移植时需要注意：

- 保留 M5 当前 `GPIO35 / ADC1_CHANNEL_7` 的电池采样配置；
- 不直接覆盖 `board_config.h`；
- 不盲目套用上游电池低压判断变化，避免 M5 上 ADC 异常读数导致显示误判；
- 设置菜单中增加“电池显示”一行，在“电量校准”附近更合理。

建议模式：

| 模式 | 显示 |
| --- | --- |
| 图标 | 保持现有三色图标 |
| 百分比 | 例如 `87%` |
| 电压 | 例如 `4.08V` |

### 2. QTH / 归属地显示

建议跟进，但默认关闭，优先级中。

上游通过 Grid 信息异步查询地址，并在当前呼号下方显示 QTH。这个功能对日常观察通联有价值，但对 M5 的 320x240 屏幕、网络请求量和内存占用都有影响。

M5 分支移植建议：

- 默认关闭；
- 设置菜单增加 `QTH显示` 开关；
- 只在事件数据有 Grid 时请求；
- 对同一 Grid 做缓存，避免重复请求；
- 屏幕空间不足时只显示简短地址；
- 不影响点名、通联状态、LED 提示和音频实时性。

需要重点验证：

- QTH 关闭时不能产生额外网络请求；
- FMO 断线或 WiFi 弱时不能卡住 UI；
- 当前呼号、状态文本、QTH 文本不能重叠；
- 中文地址字体正常显示；
- 接口失败时显示保持安静，不频繁刷错误提示。

### 3. 蜂鸣器相关功能

明确不跟进，除非后续单独为 M5 内置扬声器重新设计。

原因：

- 上游蜂鸣器面向外接蜂鸣器；
- 上游使用的蜂鸣器 GPIO 与 M5 硬件引脚存在冲突风险；
- M5 分支已经使用内置音频和侧边 LED 做提醒；
- 直接合并 `gpio_ctrl.c/.h`、蜂鸣器设置项、开机音和省电音效，可能破坏当前显示或音频链路。

后续如果需要声音提示，应单独基于 M5 的 DAC / 扬声器链路实现，不复用上游外接蜂鸣器方案。

## 三、M5 硬件传感器可扩展功能

### 晃动切换上一个 / 下一个活跃中继

可行，建议作为 M5 分支特色功能加入，但第一版应默认关闭。

M5 Core / Fire 的部分型号带有 IMU 加速度传感器。可以新增一个低频传感器任务，检测明显的左右晃动手势，并映射为“切换到上一个 / 下一个活跃中继”。

推荐交互：

| 手势 | 行为 |
| --- | --- |
| 向左晃一下 | 切换到上一个活跃中继 |
| 向右晃一下 | 切换到下一个活跃中继 |
| 连续晃动 | 只响应第一次，进入短暂冷却 |

建议策略：

- 设置菜单增加 `晃动切换中继` 开关，默认关闭；
- 只在主屏或待机屏响应，设置页、WiFi 输入页、FMO 输入页不响应；
- 正在通联、正在发声、FMO 未连接、站点列表为空时不响应；
- 每次触发后至少冷却 2~3 秒，避免拿起设备时误切换多次；
- 触发后在底部状态栏显示“切换上一个中继”或“切换下一个中继”；
- 如果当前机型未检测到 IMU，设置项显示“不可用”或隐藏。

实现方向：

1. 在 `board_config.h` 增加 IMU 硬件开关和 I2C 引脚定义；
2. 新增 `app_motion.c/.h`，负责 IMU 初始化、采样、手势识别和冷却；
3. 根据实际 M5 型号识别 MPU6886、MPU9250 或 SH200Q 等 IMU；
4. 从 `station_parser` 的站点缓存中寻找当前站点前后相邻项；
5. 调用现有 `audio_ws_station_set_current(uid)` 完成切换；
6. 切换后延迟请求 `audio_ws_station_get_current()`，刷新 UI。

需要验证：

- 不同 M5 型号的 IMU 芯片和 I2C 地址可能不同；
- 当前分支尚未启用 IMU/I2C，需确认不会与 LCD、SD、音频、按键冲突；
- 手势阈值要足够高，避免正常拿起、放下、移动设备时误切换；
- 切换中继是网络操作，必须避免在断网或 WebSocket 未连接时发送；
- 如果站点列表只加载一页，需要先决定“上一个/下一个”的范围是当前缓存、收藏中继，还是所有活跃中继。

## 四、当前设置功能调整需求

### 1. 取消节能模式 / 休眠功能

建议取消或默认彻底关闭，优先级高。

当前节能模式会执行一组较重的动作：停止 WebSocket、停止 WiFi、进入省电时钟页、降低背光；退出时再恢复背光、重启 WiFi，并等待 WiFi 获取 IP 后恢复 WebSocket。这个流程在实际 M5 上如果任一状态没有恢复完整，就容易表现为黑屏、断联或“像死机”。

建议第一版处理方式：

- 移除主屏左下角节能按钮；
- 移除设置菜单中的“节能模式”开关；
- 禁用手动节能入口；
- 禁用自动待机进入省电；
- 保留低电量保护逻辑的讨论空间，但不要再停止 WiFi / WebSocket；
- LED 节能特殊处理可以随功能移除而简化。

如果仍希望保留低亮度时钟页，可把它改成“屏保”而不是“省电”：只显示时钟/呼号、降低背光，不停 WiFi、不停 WebSocket、不影响通联。

需要验证：

- 长时间运行不再因为省电模式断 WiFi；
- 通联中不会进入任何会停止 WebSocket 的状态；
- 原有 LED 通联提示不会被省电状态卡住；
- 设置菜单中不再出现无用的节能开关。

### 2. 本机呼号改为从 FMO 同步

建议取消手动设置入口，改为优先从 FMO 同步获取，优先级高。

当前本机呼号用于：

- 省电/待机页显示；
- LED 逻辑中判断“本机呼号通联”为红灯；
- 新呼号提醒中排除本机呼号。

如果取消设置项，仍必须保留一个运行期 `owner_callsign`，只是来源从“用户手动输入”改为“FMO 同步”。建议来源优先级：

1. FMO 返回的本机呼号；
2. NVS 中上次同步成功保存的本机呼号；
3. 编译期默认值 `BH1JSS` 作为兜底。

实现方向：

- 移除设置菜单“本机呼号”输入页；
- 保留 `owner_callsign` 字段用于内部判断和显示；
- 在 FMO 连接成功后请求或解析本机信息；
- 同步到本机呼号后保存到 NVS；
- LED 判断继续使用当前 `owner_callsign`，避免自己的呼号触发新呼号动画。

需要先确认 FMO 当前 WebSocket 或 HTTP API 中哪个消息包含本机呼号。若没有现成字段，需要抓一次 FMO 返回 JSON 或查原服务端接口。

### 3. 获取并显示 FM 通讯频率

建议跟随“本机信息 / 当前站点信息同步”一起做，优先级中。

当前分支的站点解析只保存 `uid` 和 `name`，QSO 事件只解析 `callsign/grid/isSpeaking`，尚未解析频率字段。若 FMO 返回当前频率，可在主屏站点区域附近显示，例如：

```text
当前中继  北京-xxx
频率      438.500 MHz
```

候选数据来源：

- 当前站点 `getCurrentResponse` 中的 `frequency` / `freq` / `rxFreq` / `txFreq` 字段；
- 站点列表项中的频率字段；
- FMO 用户/设备状态接口中的当前 FM 频率字段。

实现方向：

- 扩展 `station_item_t`，增加频率字符串字段；
- 在 `station_parser.c` 解析常见频率字段名；
- 新增 `app_ui_update_frequency()` 或把频率作为站点信息的一部分更新；
- 主屏为频率预留一行短文本，避免与当前呼号、站点名、QSO 数重叠；
- 如果 FMO 没返回频率，则隐藏该行或显示 `--`。

需要验证：

- 抓取真实 FMO JSON，确认字段名和单位；
- 频率显示不要挤占当前呼号和站点名称；
- 切换中继后频率能同步刷新；
- 多 FMO 配置切换后频率来源对应当前 FMO。

### 4. 屏幕旋转支持 90 度和 270 度

建议跟进，优先级中。

当前设置中只有 `screen_rotate_180` 布尔值，只能在“正常 / 180 度”之间切换。若要支持竖屏或不同安装方向，应改为旋转枚举：

| 旋转值 | 显示方向 |
| --- | --- |
| `0` | 正常横屏 |
| `90` | 顺时针 90 度 |
| `180` | 倒置横屏 |
| `270` | 顺时针 270 度 |

实现方向：

- 将 `screen_rotate_180` 替换或迁移为 `screen_rotation`；
- 设置菜单“屏幕旋转”从二选一改为循环切换 `0° -> 90° -> 180° -> 270°`；
- LVGL 显示驱动层根据旋转值调整 flush 坐标和屏幕宽高；
- 三按键方向在 90/270 度时需要重新确认 A/C 的“上/下”或“左/右”体验；
- 所有主屏、设置页、软键盘、站点弹窗需要按 240x320 竖屏重新验证布局。

需要注意：

- 90/270 度不是简单改 `MADCTL` 就一定够用，LVGL 分辨率、布局尺寸和输入方向也要一起适配；
- 当前 UI 大量按 320x240 横屏设计，竖屏下可能需要专门布局；
- 第一版可以先支持驱动旋转和基础页面可用，再逐步优化竖屏排版。

## 五、建议实施顺序

1. 多地点连接配置：先完成 5 个配置槽、当前配置切换、FMO Host[:Port] 校验和旧设置迁移。
2. 取消节能模式 / 休眠功能：移除入口，避免停止 WiFi 和 WebSocket 造成假死。
3. 本机呼号自动同步：取消手动设置，FMO 获取优先，NVS 缓存兜底。
4. 当前频率显示：先抓取真实 FMO JSON，确认字段后解析显示。
5. 屏幕旋转 90/270：先完成设置迁移和驱动旋转，再评估竖屏 UI 排版工作量。
6. 电池显示方式：移植上游“图标 / 百分比 / 电压”思路，但保留 M5 电池硬件配置。
7. 晃动切换中继：先做 IMU 探测和默认关闭的手势开关，再小范围验证误触发率。
8. QTH 显示：默认关闭，小范围移植并验证屏幕布局和网络稳定性。
9. 仅在确认硬件方案后，再考虑 M5 自己的声音提示；上游蜂鸣器代码继续不动。
